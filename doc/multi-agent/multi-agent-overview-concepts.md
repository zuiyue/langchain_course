# LangChain Multi-Agent Overview 核心概念教程


---

## 一、什么是 Multi-Agent 系统？

### 1.1 从"一个人干所有事"到"团队分工协作"

想象你在经营一家公司：

```
单智能体（一个人干所有事）:
  CEO 同时负责：
    - 客服接待
    - 财务核算
    - 技术研发
    - 市场营销
    - 法律合规
    
  问题：
    ❌ 上下文过载：每个领域的知识都塞进大脑
    ❌ 注意力分散：每次要决定"现在该做什么"
    ❌ 难以扩展：一个人能力有上限
    ❌ 无法并行：同一时间只能做一件事

多智能体（团队分工协作）:
  CEO（协调者）
    ├── 客服专员（只处理客户问题）
    ├── 财务专员（只处理账目）
    ├── 技术专家（只写代码）
    ├── 市场专员（只做营销）
    └── 法务顾问（只审合同）
    
  优势：
    ✅ 上下文隔离：每个专家只知道自己领域的事
    ✅ 专注高效：每个智能体上下文精简
    ✅ 易于扩展：新增一个专员即可
    ✅ 可以并行：多个专员同时工作
```

**核心定义**：多智能体系统通过**协调专用组件**（智能体/工具/上下文）来处理复杂工作流。其设计核心是**上下文工程（Context Engineering）**——精确控制每个智能体能访问的信息，避免知识过载，实现按需、精准的信息表面化。

### 1.2 什么时候需要 Multi-Agent？

```
⚠️ 重要提醒：并非所有复杂任务都需要多智能体！

单智能体就够用的场景：
  ✅ 任务聚焦在一个领域
  ✅ 工具数量 < 10 个
  ✅ 提示词长度适中
  ✅ 不需要严格的流程控制

需要多智能体的场景：
  🔹 单智能体工具过多导致路由决策变差
  🔹 任务依赖长提示词与专业领域工具
  🔹 需要严格按条件解锁顺序执行
  🔹 需要分布式开发（不同团队维护不同模块）
  🔹 需要大规模并行化加速产出
```

## 二、为什么需要 Multi-Agent？

### 2.1 解决三大核心问题

```
┌─────────────────────────────────────────────────────────┐
│              多智能体解决的三大核心问题                     │
│                                                         │
│  1. 上下文管理（Context Management）                      │
│     避免海量知识挤占模型上下文窗口                         │
│     实现按需、精准的信息表面化                             │
│                                                         │
│  2. 分布式开发（Distributed Development）                  │
│     允许不同团队独立开发和维护各自的能力模块                 │
│     通过清晰边界组合成大系统                               │
│                                                         │
│  3. 并行化（Parallelization）                             │
│     为子任务生成专用工作器                                 │
│     支持并发执行以加速结果产出                             │
└─────────────────────────────────────────────────────────┘
```

### 2.2 上下文管理的挑战

```
单智能体上下文过载：
  用户："帮我分析一下这家公司：市场地位、财务状况、技术实力、法律风险"
  
  智能体大脑中需要同时加载：
    - 市场分析框架（5K token）
    - 财务分析模型（5K token）
    - 技术评估标准（5K token）
    - 法律合规知识（5K token）
    - ... 总共 20K+ token！
  
  结果：
    ❌ 上下文窗口被占满
    ❌ 模型注意力分散
    ❌ 回答质量下降

多智能体上下文隔离：
  协调者将任务拆分：
    - 市场智能体：只加载市场分析框架（5K token）
    - 财务智能体：只加载财务模型（5K token）
    - 技术智能体：只加载技术标准（5K token）
    - 法务智能体：只加载法律知识（5K token）
  
  结果：
    ✅ 每个智能体上下文精简
    ✅ 专注度高，回答质量好
    ✅ 总消耗更少（隔离设计更高效）
```

## 三、架构设计核心

### 3.1 三大设计维度

```
┌─────────────────────────────────────────────────────────┐
│              多智能体架构设计三维度                        │
│                                                         │
│  1. 路由分发（Routing）                                  │
│     请求如何分类和分发到正确的智能体？                      │
│     - 由谁决定？（LLM 分类 / 规则匹配 / 状态驱动）         │
│     - 何时决定？（请求入口 / 执行过程中）                  │
│                                                         │
│  2. 状态管理（State Management）                          │
│     控制权如何转移和保持？                                 │
│     - 无状态：每次调用独立，强隔离                         │
│     - 有状态：上下文持久，重复请求成本低                    │
│                                                         │
│  3. 上下文隔离（Context Isolation）                       │
│     专用上下文何时加载？                                   │
│     - 预加载：启动时就加载所有上下文                       │
│     - 按需加载：需要时才加载特定上下文                      │
│     - 动态卸载：不使用时释放上下文                         │
└─────────────────────────────────────────────────────────┘
```

### 3.2 无状态 vs 有状态权衡

```
┌─────────────────────────────────────────────────────────┐
│              无状态 vs 有状态架构对比                       │
│                                                         │
│  无状态架构                         有状态架构             │
│  (Subagents / Router)              (Handoffs / Skills)   │
│                                                         │
│  ✅ 强上下文隔离                    ✅ 重复请求成本低       │
│  ✅ 适合大上下文隔离                ✅ 可节省 40-50% 调用  │
│  ✅ 适合高并发                      ✅ 上下文持久           │
│                                                         │
│  ❌ 每次需重新加载上下文            ❌ 上下文可能污染       │
│  ❌ 重复请求成本高                  ❌ 隔离性较弱           │
│                                                         │
│  适用：一次性任务、多领域查询         适用：连续对话、       │
│                                    用户交互密集型          │
└─────────────────────────────────────────────────────────┘
```

## 四、五大模式详解

LangChain 定义了五种多智能体模式，各有适用场景：

### 4.1 模式一：Subagents（子智能体）

**工作机制**：主智能体将子智能体作为**工具**协调。所有路由经主智能体中转。

```
┌─────────────────────────────────────────────────────────┐
│                    Subagents 架构                          │
│                                                         │
│                      用户输入                             │
│                         │                               │
│                         ▼                               │
│                  ┌─────────────┐                         │
│                  │  主智能体    │                         │
│                  │  (协调者)    │                         │
│                  └──────┬──────┘                         │
│                         │ 路由决策                        │
│            ┌────────────┼────────────┐                   │
│            ▼            ▼            ▼                   │
│     ┌──────────┐ ┌──────────┐ ┌──────────┐              │
│     │子智能体 A │ │子智能体 B │ │子智能体 C │              │
│     │(市场)    │ │(财务)    │ │(技术)    │              │
│     └─────┬────┘ └─────┬────┘ └─────┬────┘              │
│           │            │            │                    │
│           └────────────┼────────────┘                    │
│                        ▼                                 │
│                  结果汇总返回                             │
│                         │                               │
│                         ▼                               │
│                      用户输出                             │
└─────────────────────────────────────────────────────────┘
```

**特点**：

| 特性 | 说明 |
|------|------|
| **控制权** | 主智能体始终掌握控制权 |
| **通信方式** | 主智能体通过工具调用下发任务 |
| **上下文** | 子智能体执行后返回结果，上下文隔离 |
| **调用开销** | 每次需多 1 次调用（结果需返回主节点） |
| **Token 效率** | 因隔离设计，单次请求约 ~9K token |

**适用场景**：多领域查询、一次性任务、需要强隔离的场景

**类比理解**：
```
Subagents = 项目经理模式
  项目经理（主智能体）接到任务后：
  - 分析需要什么专业能力
  - 分配给对应的专家（子智能体）
  - 收集专家的报告
  - 汇总成最终答案交给客户
```

### 4.2 模式二：Handoffs（交接）

**工作机制**：基于**状态**动态改变行为。工具调用更新状态变量，触发路由/配置变更，切换智能体或调整其工具/提示词。

```
┌─────────────────────────────────────────────────────────┐
│                    Handoffs 架构                           │
│                                                         │
│   用户："帮我退款，然后换个新手机"                         │
│      │                                                  │
│      ▼                                                  │
│   ┌──────────────┐                                      │
│   │  客服智能体   │  处理退款请求                         │
│   │  (当前活跃)   │                                      │
│   └──────┬───────┘                                      │
│          │ 工具调用：transfer_to_sales()                  │
│          ▼                                              │
│   ┌──────────────┐                                      │
│   │  销售智能体   │  ← 状态更新，切换活跃智能体             │
│   │  (当前活跃)   │  处理换机请求                         │
│   └──────────────┘                                      │
└─────────────────────────────────────────────────────────┘
```

**特点**：

| 特性 | 说明 |
|------|------|
| **控制权** | 动态转移，基于状态变量 |
| **通信方式** | 通过共享状态变量传递控制权 |
| **上下文** | 持久保持，智能体切换时保留历史 |
| **调用开销** | 重复请求时可节省 40-50% 调用 |
| **Token 效率** | 多领域任务可达 ~15K token（累加） |

**适用场景**：连续对话、用户交互密集型、需要状态保持的场景

**类比理解**：
```
Handoffs = 电话转接模式
  客服接到电话：
  - 先处理自己的部分（退款）
  - 然后转给销售（"请稍等，我帮您转接销售部"）
  - 销售接手后继续对话
  - 客户感觉在和"同一个人"对话
```

**⚠️ 限制**：Handoffs 在处理多领域任务时只能**串行**执行，无法利用并行工具调用，效率最低。

### 4.3 模式三：Skills（技能）

**工作机制**：按需加载专用提示词和知识。**单个智能体始终保持控制权**，动态挂载上下文。

```
┌─────────────────────────────────────────────────────────┐
│                     Skills 架构                            │
│                                                         │
│   单个智能体，但可以根据任务动态切换"技能包"：              │
│                                                         │
│   初始状态:                                              │
│   ┌──────────────────┐                                  │
│   │    通用智能体     │                                  │
│   │  技能: 基础对话   │                                  │
│   └──────────────────┘                                  │
│                                                         │
│   用户问技术问题:                                         │
│   ┌──────────────────┐                                  │
│   │    通用智能体     │  动态挂载                          │
│   │  技能: + 技术文档 │  ──► 技术专家模式                 │
│   │        + API 手册 │                                  │
│   └──────────────────┘                                  │
│                                                         │
│   用户问财务问题:                                         │
│   ┌──────────────────┐                                  │
│   │    通用智能体     │  动态切换                          │
│   │  技能: + 财务模型 │  ──► 财务专家模式                 │
│   │        + 会计准则 │                                  │
│   └──────────────────┘                                  │
└─────────────────────────────────────────────────────────┘
```

**特点**：

| 特性 | 说明 |
|------|------|
| **控制权** | 单个智能体始终持有 |
| **通信方式** | 内部动态注入/切换专用 Prompt 和知识库 |
| **上下文** | 持续累加（多领域任务可达 ~15K token） |
| **调用开销** | 重复请求时因状态保留可节省加载调用 |
| **适用任务** | 任务简单聚焦、单领域多技能 |

**适用场景**：单智能体足够但需要多领域知识、任务简单聚焦

**类比理解**：
```
Skills = 瑞士军刀模式
  一把刀（单个智能体）始终在手
  需要时展开不同的工具：
  - 拧螺丝时展开螺丝刀头
  - 开瓶时展开开瓶器
  - 剪纸时展开剪刀
  刀还是那把刀，但功能在变
```

### 4.4 模式四：Router（路由器）

**工作机制**：路由步骤对输入进行分类，分发至一个或多个专用智能体，最终将结果合成统一响应。

```
┌─────────────────────────────────────────────────────────┐
│                     Router 架构                            │
│                                                         │
│                     用户输入                              │
│                        │                                │
│                        ▼                                │
│                 ┌─────────────┐                          │
│                 │   路由器     │  LLM 分类器               │
│                 │ (分类意图)   │                          │
│                 └──────┬──────┘                          │
│                        │ 分类结果                         │
│           ┌────────────┼────────────┐                    │
│           ▼            ▼            ▼                    │
│    ┌──────────┐ ┌──────────┐ ┌──────────┐               │
│    │智能体 A   │ │智能体 B   │ │智能体 C   │    ← 并行    │
│    │(技术)    │ │(财务)    │ │(市场)    │               │
│    └─────┬────┘ └─────┬────┘ └─────┬────┘               │
│          │            │            │                     │
│          └────────────┼────────────┘                     │
│                       ▼                                  │
│                 ┌─────────────┐                          │
│                 │  结果合成    │                          │
│                 └──────┬──────┘                          │
│                        │                                │
│                        ▼                                │
│                     用户输出                              │
└─────────────────────────────────────────────────────────┘
```

**特点**：

| 特性 | 说明 |
|------|------|
| **控制权** | 路由器先分类，再分发 |
| **通信方式** | LLM 作为分类器解析意图 |
| **上下文** | 各智能体独立，强隔离 |
| **并行能力** | 支持并行调用多个智能体 |
| **适用任务** | 多领域一次性查询 |

**适用场景**：需要并行处理多领域任务、一次性综合查询

**类比理解**：
```
Router = 医院分诊台
  病人来描述症状
  分诊护士（路由器）判断：
  - "你去心内科"
  - "你去骨科"
  - "你需要同时看两个科"
  各科室医生（智能体）独立诊断
  最后汇总成完整的诊断报告
```

### 4.5 模式五：Custom Workflow（自定义工作流）

**工作机制**：基于 **LangGraph** 构建混合确定性逻辑与智能体行为的定制流程，可嵌入其他模式作为节点。

```
┌─────────────────────────────────────────────────────────┐
│                 Custom Workflow 架构                       │
│                                                         │
│   使用 LangGraph 构建任意复杂的流程：                       │
│                                                         │
│   输入 ──► [节点A: 数据验证] ──► [节点B: Router 分类]     │
│                                      │                   │
│                        ┌─────────────┼─────────────┐     │
│                        ▼             ▼             ▼     │
│                   [节点C:        [节点D:        [节点E:   │
│                    技术Agent]     财务Agent]     法务Agent]│
│                        │             │             │     │
│                        └─────────────┼─────────────┘     │
│                                      ▼                   │
│                         [节点F: 结果验证 + 合成]           │
│                                      │                   │
│                                      ▼                   │
│                                   输出                    │
│                                                         │
│   可以嵌入其他模式作为节点：                                │
│   - 节点 C 内部可以使用 Subagents                        │
│   - 节点 D 内部可以使用 Handoffs                         │
│   - 节点 E 内部可以使用 Skills                           │
└─────────────────────────────────────────────────────────┘
```

**特点**：

| 特性 | 说明 |
|------|------|
| **灵活性** | 最高，可构建任意复杂流程 |
| **控制权** | 完全自定义（确定性 + 智能体混合） |
| **复杂度** | 最高，需要 LangGraph 编排能力 |
| **适用场景** | 复杂企业流程、需要严格控制的场景 |

**类比理解**：
```
Custom Workflow = 工厂流水线
  你可以设计任意复杂的生产线：
  - 哪些步骤是自动的（确定性逻辑）
  - 哪些步骤需要人工判断（智能体）
  - 步骤之间的条件和分支
  - 异常处理和质量检查
```

## 五、五大模式对比

### 5.1 核心特性对比

| 模式 | 控制权 | 状态 | 并行 | 上下文隔离 | 适用场景 |
|------|--------|------|------|-----------|---------|
| **Subagents** | 主智能体 | 无状态 | ✅ | 强 | 多领域查询 |
| **Handoffs** | 动态转移 | 有状态 | ❌ | 弱 | 连续对话 |
| **Skills** | 单智能体 | 有状态 | ❌ | 中 | 简单聚焦 |
| **Router** | 路由器 | 无状态 | ✅ | 强 | 并行查询 |
| **Custom** | 自定义 | 自定义 | 自定义 | 自定义 | 复杂流程 |

### 5.2 性能对比

```
单次请求（One-shot）Token 消耗：
  Subagents:  ~9K token  ← 最隔离，效率高
  Router:     ~9K token  ← 类似 Subagents
  Skills:     ~11K token ← 上下文累加
  Handoffs:   ~15K token ← 上下文持续累加

多领域任务（Multi-domain）Token 消耗：
  Subagents:  ~9K token  ← 每个子智能体独立，隔离好
  Router:     ~9K token  ← 并行，隔离好
  Skills:     ~15K token ← 上下文持续累加
  Handoffs:   ~15K token ← 串行，上下文累加

重复请求（Repeated）调用次数：
  Handoffs:   节省 40-50% ← 状态保持，无需重新加载
  Skills:     节省 40-50% ← 状态保持，无需重新加载
  Subagents:  无节省      ← 每次重新加载
  Router:     无节省      ← 每次重新路由
```

### 5.3 选型决策树

```
需要多智能体吗？
  │
  ├── 任务简单、工具 < 10 个 ──► 用单智能体 + Skills
  │
  └── 需要多智能体
       │
       ├── 需要并行处理多领域？
       │    ├── 是 ──► Router（一次性查询）或 Subagents
       │    └── 否 ──► 继续
       │
       ├── 需要保持对话状态？
       │    ├── 是 ──► Handoffs（连续交互）
       │    └── 否 ──► 继续
       │
       ├── 需要强上下文隔离？
       │    ├── 是 ──► Subagents
       │    └── 否 ──► Skills
       │
       └── 流程非常复杂？
            └── 是 ──► Custom Workflow (LangGraph)
```

## 六、智能体之间的通信和协调

### 6.1 四种通信机制

```
┌─────────────────────────────────────────────────────────┐
│              智能体通信机制全景图                          │
│                                                         │
│  1. 工具调用（Tool Calling）                              │
│     Subagents 模式                                       │
│     主智能体通过工具调用下发任务                           │
│     子智能体执行后返回结果                                │
│                                                         │
│  2. 状态变量（State Variables）                           │
│     Handoffs 模式                                        │
│     通过共享状态变量传递控制权                             │
│     工具调用更新状态，触发切换                            │
│                                                         │
│  3. 动态注入（Dynamic Injection）                         │
│     Skills 模式                                          │
│     单智能体内部动态挂载上下文                             │
│     无需跨智能体跳转                                     │
│                                                         │
│  4. 分类分发（Classification & Dispatch）                 │
│     Router 模式                                          │
│     LLM 作为分类器解析意图                               │
│     并行或串行调用目标智能体                              │
└─────────────────────────────────────────────────────────┘
```

### 6.2 并行工具调用

所有模式都支持**并行工具调用**以提升多领域任务处理效率：

```
用户："对比分析苹果、微软、谷歌三家公司"

Subagents 并行：
  主智能体同时调用：
    - 分析苹果的子智能体 ─┐
    - 分析微软的子智能体 ─┼── 同时执行
    - 分析谷歌的子智能体 ─┘
  
Router 并行：
  路由器同时分发到：
    - 苹果分析智能体 ─┐
    - 微软分析智能体 ─┼── 同时执行
    - 谷歌分析智能体 ─┘
```

## 七、使用场景与最佳实践

### 7.1 场景匹配指南

| 需求 | 推荐模式 | 原因 |
|------|----------|------|
| 分布式开发 | Subagents / Custom | 模块边界清晰，独立维护 |
| 强并行处理 | Router / Subagents | 支持并发执行 |
| 多跳推理 | Subagents / Router | 多步骤分解 |
| 状态保持 | Handoffs / Skills | 上下文持久 |
| 直接用户交互 | Handoffs / Skills | 连续对话体验好 |
| 任务简单聚焦 | Skills | 单智能体足够 |
| 复杂企业流程 | Custom Workflow | 灵活性最高 |

### 7.2 最佳实践

#### 实践 1：做好上下文工程

精确控制每个智能体能访问的信息，避免知识过载。

```
原则：
  ✅ 每个智能体只加载必要的知识
  ✅ 按需加载，不用时不加载
  ✅ 用完后释放（无状态模式自动释放）
```

#### 实践 2：避免过度设计

单智能体足够时不强行升级。

```
检查清单：
  □ 工具数量 < 10 个？
  □ 提示词长度适中？
  □ 不需要严格流程控制？
  □ 不需要分布式开发？
  
  如果都是"是" → 用单智能体！
```

#### 实践 3：根据请求频率选型

```
单次/低频请求：
  ──► 无状态模式（Subagents / Router）
  原因：无需保持状态，每次独立处理

重复/高频请求：
  ──► 有状态模式（Handoffs / Skills）
  原因：状态保持可节省 40-50% 调用
```

#### 实践 4：混合使用模式

模式可以混合使用，发挥各自优势。

```
混合示例：
  Subagents 调用 Router：
    主智能体 ──► 子智能体（技术分析）
                        │
                        └──► 内部使用 Router 分类：
                                 ├── 前端分析智能体
                                 └── 后端分析智能体
  
  Handoffs 内部使用 Skills：
    客服智能体 ──► 切换到"退款技能"
                 ──► 切换到"换货技能"
                 ──► transfer_to_sales() ──► 销售智能体
```

#### 实践 5：监控性能和成本

跟踪每个智能体的调用次数、Token 消耗和延迟。

```
关键指标：
  - 每个智能体的调用次数
  - 每次请求的 Token 消耗
  - 端到端延迟
  - 并行度利用率
  
优化方向：
  - 减少不必要的上下文加载
  - 提高并行度
  - 合并相似能力的智能体
```

### 8.1 模式可组合性

```
模式组合示例：

┌─────────────────────────────────────────────────────────┐
│                  混合架构示例                              │
│                                                         │
│   Router（入口分类）                                     │
│     ├── 技术问题 ──► Subagents                          │
│     │                  ├── 前端子智能体（内部用 Skills）   │
│     │                  └── 后端子智能体                   │
│     │                                                    │
│     ├── 财务问题 ──► Handoffs                           │
│     │                  ├── 审计智能体                     │
│     │                  └── 报税智能体                     │
│     │                                                    │
│     └── 复杂问题 ──► Custom Workflow                    │
│                        ├── 数据收集节点                   │
│                        ├── 分析节点                       │
│                        └── 合成节点                       │
└─────────────────────────────────────────────────────────┘
```

### 8.2 调用开销分析

```
Subagents 额外开销：
  每次调用多 1 次调用（结果需返回主节点）
  请求 ──► 主智能体 ──► 子智能体 ──► 主智能体 ──► 响应
           (1)          (2)           (3)          (4)

Handoffs/Skills 节省：
  重复请求时因状态保留可省去路由/加载调用
  第一次请求 ──► 加载上下文（开销大）
  第二次请求 ──► 直接用已有上下文（开销小）
```

### 8.3 串行瓶颈

```
Handoffs 的串行限制：
  ❌ 无法并行调用多个智能体
  
  用户："同时分析技术和财务"
  Handoffs 执行：
    技术智能体执行 ──► 完成后 ──► 财务智能体执行
  
  如果改用 Subagents 或 Router：
    技术智能体 ─┐
                ├── 同时执行，速度快
    财务智能体 ─┘
```

## 九、常见错误与陷阱

### 陷阱 1：盲目追求多智能体

**现象**：简单任务也使用多智能体，导致延迟与成本激增。

```
错误做法：
  用户："今天天气怎么样？"
  系统：Router ──► 天气智能体 ──► 合成 ──► 响应
  （过度设计！单智能体直接回答即可）

正确做法：
  用户："今天天气怎么样？"
  系统：单智能体 ──► 响应
```

### 陷阱 2：忽略上下文隔离

**现象**：所有智能体共享大量上下文，导致 Token 浪费。

```
错误做法：
  每个子智能体都收到完整的 20K token 上下文
  包括它根本不需要的知识

正确做法：
  技术智能体只收到技术相关上下文（~3K token）
  财务智能体只收到财务相关上下文（~3K token）
```

### 陷阱 3：未根据请求频率选型

**现象**：高频重复请求使用无状态模式，成本极高。

```
错误做法：
  客服对话（同一用户连续问 10 个问题）
  使用 Subagents（每次重新加载上下文）
  
  成本：10 次 × 完整加载 = 高昂

正确做法：
  客服对话使用 Handoffs 或 Skills
  保持状态，重复请求成本降低 40-50%
```

### 陷阱 4：Handoffs 用于需要并行的场景

**现象**：Handoffs 只能串行，处理多领域任务时效率最低。

```
错误做法：
  用户："同时分析市场、财务、技术"
  使用 Handoffs：
    市场分析 ──► 完成 ──► 财务分析 ──► 完成 ──► 技术分析
    （总时间 = 3 倍单任务时间）

正确做法：
  使用 Subagents 或 Router：
    市场分析 ─┐
    财务分析 ─┼── 同时执行
    技术分析 ─┘
    （总时间 ≈ 1 倍单任务时间）
```

## 十、总结：Multi-Agent 的本质

| 概念 | 比喻 | 作用 |
|------|------|------|
| **Multi-Agent** | 团队分工 | 专用组件处理复杂流程 |
| **Context Engineering** | 信息过滤 | 精确控制每个智能体的知识 |
| **Subagents** | 项目经理 | 主智能体协调子工具 |
| **Handoffs** | 电话转接 | 基于状态动态切换 |
| **Skills** | 瑞士军刀 | 单智能体动态挂载技能 |
| **Router** | 医院分诊 | 分类意图并行分发 |
| **Custom** | 工厂流水线 | LangGraph 编排任意流程 |
| **无状态** | 每次重启 | 强隔离，重复成本高 |
| **有状态** | 保持记忆 | 重复成本低，上下文累加 |

**Multi-Agent 的本质**是**通过上下文工程实现专业化分工**。它不是为复杂而复杂，而是为了解决单智能体无法高效处理的问题——上下文过载、路由困难、无法并行、难以分布式开发。

```
一句话总结：

多智能体系统的核心不是"多用几个智能体"，而是
"做好上下文工程"——精确控制每个智能体访问的信息，
实现按需、精准的信息表面化，避免过度设计。
```
